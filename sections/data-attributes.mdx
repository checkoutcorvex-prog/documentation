---
title: "Data Attributes: How Sections Wire Up to JavaScript"
description: "Required and optional data-* attributes in Corvex Store sections — identifiers, typed options, element selectors, state flags, and runtime debug attributes."
---

Corvex Store sections use `data-*` HTML attributes for all JavaScript concerns: identification, element targeting, actions, state, and blocks. This is deliberate — `class` is reserved exclusively for CSS. Following this convention is what makes lifecycle hooks, editor integration, and event delegation work correctly.

<Warning>
If your JavaScript logic depends on a `class`, refactor to `data-*` before writing the next test.
</Warning>

## Required attributes on the root element

### `data-section-type`

Identifies the **type** of the section. This is the key used to look up the init function in the registry (`CorvexSections.register('<type>', ...)`).

```liquid
<section data-section-type="hero-banner" ...>
```

- Must be kebab-case (`hero-banner`, not `heroBanner`).
- Must be **unique per type**. Never register two functions for the same type.
- If no function is registered for the type, the element is silently ignored (no error).

### `data-section-id`

Identifies a **specific instance** of a section on the page. This is the key the editor uses to link the preview to the right panel, and the key you use to scope CSS.

```liquid
<section data-section-id="{{ section.id }}" ...>
```

- Generated by the backend/editor — always use `{{ section.id }}`.
- If the same section appears twice on a page, each instance gets a different `section.id`.
- `CorvexSections` detects duplicates with the same `data-section-id` and **aborts** initialization of the extra copy, preventing duplicate event handlers.

## Optional attributes on the root element

Any `data-*` attribute that is **not** one of the reserved names becomes a typed `option` automatically when you use [`BaseSection`](/sections/base-section):

| HTML attribute | Key in `this.options` | Resulting type |
|---|---|---|
| `data-autoplay-speed="5000"` | `autoplaySpeed` | `5000` (number) |
| `data-autoplay="true"` | `autoplay` | `true` (boolean) |
| `data-label="See more"` | `label` | `"See more"` (string) |
| `data-endpoint="/api/x"` | `endpoint` | `"/api/x"` (string) |

Type coercion is automatic:
- `"true"` / `"false"` → boolean
- `"-12"`, `"3.14"` → number
- Anything else → string

```liquid
<section data-section-type="slideshow"
         data-section-id="{{ section.id }}"
         data-autoplay="{{ section.settings.autoplay }}"
         data-autoplay-speed="{{ section.settings.speed }}"
         data-loop="{{ section.settings.loop }}">
```

In TypeScript:

```typescript
interface SlideshowOptions {
  autoplay: boolean;
  autoplaySpeed: number;
  loop: boolean;
}

class SlideshowSection extends BaseSection<SlideshowOptions> {
  static readonly type = 'slideshow';

  mount() {
    if (this.options.autoplay) {
      setInterval(() => this.next(), this.options.autoplaySpeed);
    }
  }
}
```

## Attributes on child elements

### `data-ref`

Marks elements that your section's JavaScript will **look up**. Replaces `querySelector` by class.

```liquid
<section data-section-type="faq" data-section-id="{{ section.id }}">
  <button data-ref="toggle-all">Expand all</button>
  <ul data-ref="question-list">
    {% for block in section.blocks %}
      <li data-ref="question" data-block-id="{{ block.id }}">
        <h3 data-ref="question-title">{{ block.settings.question }}</h3>
        <p  data-ref="question-answer">{{ block.settings.answer }}</p>
      </li>
    {% endfor %}
  </ul>
</section>
```

Usage in JavaScript with `BaseSection`:

```typescript
mount() {
  const toggle = this.ref<HTMLButtonElement>('toggle-all');  // single element
  const items  = this.refs<HTMLLIElement>('question');       // multiple elements
}
```

Without `BaseSection`, query from the section element directly:

```typescript
CorvexSections.register('faq', (el) => {
  const toggle = el.querySelector<HTMLButtonElement>('[data-ref="toggle-all"]');
});
```

<Tip>
Always search for `data-ref` **from the section element** (`el` or `this.el`) — never from `document`. Otherwise, two instances of the same section will interfere with each other.
</Tip>

### `data-action`

Marks triggers that fire a section action. Used together with event delegation:

```liquid
<button data-action="open-modal">Open</button>
<button data-action="add-to-cart" data-variant-id="{{ variant.id }}">Buy</button>
<button data-action="toggle" data-target="cart-sidebar">Open cart</button>
```

```typescript
mount() {
  this.delegate('click', {
    'open-modal':  (trigger) => this.openModal(),
    'add-to-cart': (trigger) => this.addToCart(trigger.dataset.variantId),
    'toggle':      (trigger) => this.toggle(trigger.dataset.target),
  });
}
```

### `data-state`

Communicates dynamic state that is **observable via CSS and the DOM**. Prefer this over state classes when the semantics describe a single, discrete state of the element:

```liquid
<button data-ref="cart-btn" data-state="idle">Add to cart</button>
```

```typescript
const btn = this.ref('cart-btn')!;
btn.dataset.state = 'loading';
// after response:
btn.dataset.state = 'success';
```

```css
[data-state="loading"] { opacity: 0.6; pointer-events: none; }
[data-state="success"] { background: green; }
```

### `data-block` and `data-block-id`

Mark blocks within a section — required for the editor to identify, select, and reorder them:

```liquid
{% for block in section.blocks %}
  <div data-block="{{ block.type }}" data-block-id="{{ block.id }}">
    {{ block.settings.text }}
  </div>
{% endfor %}
```

The editor uses these attributes to show the selection outline in the preview when a shopper clicks a block in the side panel.

## Reserved attributes — do not use as options

The following `data-*` attributes **never** appear in `this.options` (they are excluded from `readOptions()`):

```
data-section-type
data-section-id
data-section-initialized
data-section-failed
data-section-error
data-section-duplicate-ignored
data-ref
data-action
data-block
data-block-id
data-state
```

## Runtime-injected attributes

During initialization, `CorvexSections` may add attributes to the root element. These are useful for debugging in DevTools:

| Attribute | When it appears | Meaning |
|---|---|---|
| `data-section-initialized="true"` | After `mount()` succeeds | Section is active. |
| `data-section-failed="true"` | After `mount()` throws | Section broke on init. |
| `data-section-error="<msg>"` | With `data-section-failed` | The error message. |
| `data-section-duplicate-ignored="true"` | Duplicate detected | Instance was not initialized to prevent duplicate handlers. |

<Info>
In DevTools, search for `[data-section-failed]` to quickly identify broken sections. You can also run `window.CorvexSections.getDebugInfo()` to inspect the current runtime state.
</Info>

## Anti-patterns to avoid

```liquid
{% comment %} ❌ Class as a JS identifier {% endcomment %}
<section class="hero-banner">
  <button class="hero-btn open">...</button>
</section>
<script>
  document.querySelector('.hero-btn').addEventListener(...); {%- comment %} breaks with 2 instances {% endcomment %}
</script>

{% comment %} ❌ Inline onclick {% endcomment %}
<button onclick="doStuff()">...</button>

{% comment %} ❌ Global ID {% endcomment %}
<button id="buy-btn">...</button> {% comment %} fails if used twice on the same page {% endcomment %}

{% comment %} ❌ Section without data-section-id {% endcomment %}
<section data-section-type="banner">...</section>

{% comment %} ❌ Listener attached outside section scope {% endcomment %}
{%- comment %} document.addEventListener('click', ...); // catches clicks from other sections {% endcomment %}
```

## Quick reference

```liquid
<section
  class="banner banner-{{ section.id }}"      {%- comment %} CSS scope {% endcomment %}
  data-section-type="banner"                   {%- comment %} registry key {% endcomment %}
  data-section-id="{{ section.id }}"           {%- comment %} instance ID {% endcomment %}
  data-autoplay="{{ section.settings.auto }}"  {%- comment %} typed option {% endcomment %}
>
  <h2 data-ref="title">{{ section.settings.title }}</h2>

  <button data-action="open" data-ref="open-btn" data-state="idle">
    Open
  </button>

  {% for block in section.blocks %}
    <div data-block="{{ block.type }}" data-block-id="{{ block.id }}">
      {{ block.settings.text }}
    </div>
  {% endfor %}
</section>
```

| Attribute | Role |
|---|---|
| `class` | CSS only |
| `data-section-type` | Registry key |
| `data-section-id` | Unique instance identifier |
| `data-*` (custom) | Typed options read by `BaseSection` |
| `data-ref` | Target of `this.ref()` |
| `data-action` | Event delegation trigger |
| `data-state` | Observable dynamic state |
| `data-block` + `data-block-id` | Editor block integration |
